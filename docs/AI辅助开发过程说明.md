# AI 辅助开发过程说明

这篇东西是我给自己、也给以后翻到仓库的人留的：**这些代码基本不是我一行行敲出来的**，更多是工具和模型堆出来的；同时我想交代清楚我现在处在什么阶段、哪些是实验、哪些别当成范本。

## 我实际怎么做的

我做这个项目的时候，**全程基本没怎么手写代码**。从脚手架、写功能到联调，大头都交给 AI 编程助手和模型。我主要用的是 **Cursor**，另外 **Claude Code + minimax** 用得很多（尤其后端和全栈一块），**OpenClaw + minimax** 我也试过一部分，算是另一条路子，用来对比或补位。

比例不固定，经常是「多工具、多入口、同一个仓库」这么搅在一起，并不是只盯着一个 IDE 死磕。

## 为什么搞了 `.cursor` 和 `.claude` 里的 Skill

我不想每开一个新对话就从头讲一遍：Nest 怎么写、React 怎么写、Monorepo 里动哪几个包。所以在仓库里陆续加了一些 **Skill** 和规范向的配置：

- **`.cursor/`**：主要给 Cursor 用，跟前端、后端、Monorepo 的约定对齐。
- **`.claude/`**：主要给 Claude Code 那条线用，比如接口合同、类型、生成客户端这类流程。

说白了就是把我的**开发习惯、目录边界、质量门禁的想法**写进上下文里，让输出别每次都飘到另一个风格去。这些文件**不是终稿**，后面项目变了，我也会跟着改。

## 说实话：现在远不是「最佳实践样板」

我想讲清楚：**这不是最终形态**。仓库里有些代码我自己也知道，算不上最理想的项目实践，有的还是生成痕迹、有的先跑通再说的折中，我会一步步收、一步步完善。

另外，Skill 和规则对模型来说是**软约束**——它还是会偏、会漏，需要我下一轮对话拽回来，或者自己动手改两刀。

## 我在试什么

我想试一件很具体的事：**几乎不靠手写**，能不能从**项目初始化**一直走到**部署上线**（再加上后续维护），整条链路能不能站得住。这个我还在摸：哪些步骤可以完全交给 AI，哪些（安全、数据、发布策略之类）必须我自己盯死，我也在实践中调整。

如果你也在做类似的实验，这篇可以当对照笔记看。如果你更关心「按规范应该怎么写」，请以仓库里的 **`CLAUDE.md`、各 Skill**，以及（如果后面我补上了）**`docs/DEVELOPMENT_*.md`** 为准；**最终以代码和 CI 现在长什么样为准**。
